# ТЕХБИЗ — сайт

Многостраничный сайт компании ТЕХБИЗ («технологии для бизнеса») — комплексная автоматизация гостиничного бизнеса под ключ. Работает как одно Flask-приложение: рендерит страницы из Jinja2-шаблонов по запросу и обрабатывает форму заявки — никакой отдельной сборки и никакого PHP, ни на сервере, ни где-либо ещё.

## Назначение

Привлечение клиентов и конверсия входящего трафика в заявки на услуги автоматизации гостиниц, отелей, мини-отелей и апарт-отелей, а также отдельная продажа модуля онлайн-бронирования как самостоятельного продукта.

## Технологии

- **Приложение**: Python 3 + Flask + Jinja2 (`app.py`). Один реестр страниц (`PAGES` в `app.py`) — единственный источник правды по каждому URL, title, description и SEO-настройкам; из него же на лету генерируются `/sitemap.xml` и `/robots.txt`. Шапка/футер вынесены в шаблоны-партиалы, не дублируются по файлам.
- **Форма заявки**: маршрут `POST /send` в том же `app.py` (не отдельное приложение) — валидация, honeypot, rate-limit, отправка письма через `smtplib`.
- **Стили/скрипты**: CSS3 без фреймворков, JavaScript vanilla — лежат в `src/static/`, отдаются Flask как статика.
- **Хостинг**: один Python-процесс на весь сайт (WSGI/Passenger в ISPmanager), не PHP.

## Как запустить локально

```
pip install -r requirements.txt
python app.py
```

Приложение поднимается на `http://localhost:5000/` с чистыми URL (`/booking/`, `/1c-avtomatizaciya-otelya/`, ...), `/sitemap.xml`, `/robots.txt` — все страницы рендерятся по запросу, ничего заранее собирать не нужно.

## Чистые адреса без .html

Каждая страница — отдельный маршрут Flask с завершающим `/` (`/booking/`, `/blog/1c-avtomatizaciya-otelya-s-chego-nachat/` и т.д.), без расширения `.html`. `.htaccess` дополнительно 301-редиректит старые адреса вида `/booking.html` на `/booking/`, если они где-то проиндексированы или сохранены в закладках.

Внутренние ссылки не хардкодятся руками — `url_for('booking')`, `url_for('svc_1c')` и т.д. в шаблонах. Это не встроенный `flask.url_for`, а свой собственный Jinja-глобал (зарегистрирован через `@app.context_processor` в `app.py`), который по `id` из `PAGES` строит абсолютный путь — сделан специально простым (без relative-path арифметики), так как всё приложение живёт в одном Flask-процессе, а не выкладывается как переносимые статические файлы.

## Форма заявки

Маршрут `POST /send` в `app.py`: проверяет honeypot-поле, обязательные поля (включая согласие на обработку ПДн), рейт-лимит по IP, и отправляет письмо на `info@tex-biz.ru` через `smtplib`.

Если SMTP недоступен или сервер вернул ошибку, форма на сайте показывает понятное сообщение вместо ложного «заявка отправлена» — есть автоматический fallback на клиенте (`script.js`).

## Деплой на хостинг (reg.ru, ISPmanager)

### 0. Собрать версионированный архив для загрузки

```
bash build_release.sh
```

Пакует **ровно то, что закоммичено** в `HEAD` (`git archive`) в `releases/texbiz-deploy-v<VERSION>.zip`: весь проект (`app.py`, `passenger_wsgi.py`, `requirements.txt`, `src/templates/`, `src/static/`), кроме `.claude/` (исключён через `.gitattributes` `export-ignore` — не нужен на сервере). `<VERSION>` берётся из файла `VERSION` в корне репозитория. Некоммиченные изменения в архив не попадают — сначала закоммитить.

Каждая версия — отдельный файл в `releases/`; старые версии не перезаписываются и не удаляются (скрипт откажется собрать архив, если файл с текущей версией уже существует — сначала поднять `VERSION` и закоммитить). `releases/` в `.gitignore`, сами архивы не коммитятся, но так остаётся локальная история версий для отката.

### 1. Загрузить и распаковать

1. В ISPmanager: «Домены» / «WWW-домены» → выбрать `tex-biz.ru` → посмотреть путь к директории сайта (обычно `/var/www/<пользователь>/data/www/tex-biz.ru`).
2. Загрузить `releases/texbiz-deploy-v<VERSION>.zip` через SFTP/файловый менеджер ISPmanager и распаковать целиком прямо в эту директорию — весь проект (сайт + форма, это один Python-процесс) ложится в один корень одним шагом.
3. Проверить, что `.htaccess` реально загрузился (файловые менеджеры иногда скрывают dotfiles) и что домен уже привязан к SSL-сертификату в ISPmanager (иначе редирект на HTTPS из `.htaccess` уведёт на несуществующий сертификат).

> **Грабли (было на этом проекте):** HTTPS-редирект внутри `.htaccess` (`RewriteCond %{HTTPS} off`) вызвал `ERR_TOO_MANY_REDIRECTS` на reg.ru — за проксирующим/WAF-слоем этого хостинга `%{HTTPS}` не отражает реальный протокол запроса, и правило зацикливалось само на себя. В ISPmanager у сайта уже есть своя галочка «перенаправлять http-запросы в HTTPS» — HTTPS-редирект в `.htaccess` намеренно убран, чтобы не дублировать и не ломать эту логику; в `.htaccess` остался только редирект `www` → без `www` (он безопасен, не зависит от `%{HTTPS}`).

### 2. Python-приложение (весь сайт)

> На этом тарифе хостинг даёт только один корень для домена — отдельной директории для Python-приложений нет, но она и не нужна: весь сайт (не только форма) — один Flask-процесс, разворачивается прямо в директории домена.

1. В ISPmanager открыть раздел **«Python»** (в списке модулей слева или на странице домена) → **«Создать»**.
2. Указать: домен `tex-biz.ru`, путь приложения `/` (корень), версию Python 3. В качестве директории для файлов приложения указать директорию сайта из шага 1.1 (а не отдельную/внешнюю директорию, которую ISPmanager может предложить по умолчанию).
3. ISPmanager создаст пустое приложение с virtualenv и своим `passenger_wsgi.py`-заглушкой в этой директории. Если на шаге 1.2 уже распаковали `texbiz-deploy-v<VERSION>.zip`, в директории уже лежит настоящий `passenger_wsgi.py` — проверить, что заглушка ISPmanager перезаписана (а не наоборот).
4. Установить зависимости в virtualenv приложения. В ISPmanager обычно есть кнопка «Установить зависимости» / «pip install -r requirements.txt» на странице приложения; если её нет — через SSH: `<путь_к_venv_приложения>/bin/pip install -r requirements.txt`.
5. Если хостинг требует авторизованный SMTP, а не локальный релей — задать переменные окружения `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD` в настройках Python-приложения в ISPmanager.
6. Перезапустить приложение (кнопка «Перезапустить» / «Restart» на странице приложения в ISPmanager) — обязательно после каждой новой выкладки, иначе Passenger продолжит обслуживать старый код.
7. Свериться, что `MAIL_ENDPOINT` в `src/static/script.js` совпадает с реальным путём формы (по умолчанию в коде — `/send`; если в ISPmanager указали другой путь для приложения, поменять константу, поднять `VERSION`, закоммитить и пересобрать: `bash build_release.sh`).
8. Проверить вручную: открыть сайт, пройтись по нескольким страницам, отправить тестовую заявку через форму, убедиться, что письмо дошло на `info@tex-biz.ru`.

Точные названия кнопок могут отличаться в зависимости от версии ISPmanager — общая механика (Passenger + virtualenv + `passenger_wsgi.py`) одна и та же.

## Функциональные требования

- Адаптивность от 320px.
- Фиксированная шапка с плавным скроллом к блокам (на главной) и мобильное меню (бургер) на всех страницах.
- Анимация появления блоков при прокрутке.
- Валидация формы на клиенте (обязательные поля, формат телефона) и отправка на Flask-бэкенд, с понятным сообщением об ошибке, если он недоступен.
- Калькулятор ориентировочного бюджета (расчёт на клиенте, только на главной).

## SEO и индексация

- Уникальные `title`, `meta description`, `canonical`, Open Graph и Twitter Card — на каждой странице отдельно, задаются один раз в реестре `PAGES` (`app.py`).
- JSON-LD микроразметка: `ProfessionalService` + `FAQPage` на главной, `SoftwareApplication` на странице модуля бронирования, `Service` на страницах услуг, `Article` на статьях блога.
- `/sitemap.xml` и `/robots.txt` — Flask-роуты, генерируются на лету из того же реестра `PAGES` — не могут разойтись с реальными страницами.
- Ключевые фразы: автоматизация отеля 1С, настройка касс для гостиницы, интеграция замковой системы, настройка сервера для отеля, модуль онлайн-бронирования для отеля.
